## 1. 기본 원칙 (명시 없으면 적용)

사용자가 **추가로 지시하지 않은** UI·레이아웃·구조 변경은 하지 않는다.

**기본적으로 금지 (별도 지시 없을 때)**

| 금지 | 설명 |
|------|------|
| 위치 이동 | 요청에 없는 다른 컴포넌트·섹션의 배치 변경 |
| 추가 생성 | 새 컴포넌트, 섹션, 페이지, 에셋, API 라우트, mock 데이터 |
| 구조 개편 | 「더 이쁘게」「정리해줘」만으로 전체 헤더·그리드·반응형 재설계 |
| 부수 리팩터 | 인접 파일 스타일 통일, 네이밍 변경, dead code 정리 |
| 범위 확대 | 한 요소 수정 명목으로 연관 UI 전부 손보기 |

**허용 (요청에 포함된 범위만)**

- 사용자가 말한 **대상**의 스타일·위치·크기·문구 수정
- 버그 수정에 **필수**인 최소 래퍼/클래스 조정
- 빌드·린트 오류를 막기 위한 import 등 기계적 수정

일반 코딩 행동은 루트 [`CLAUDE.md`](../../../CLAUDE.md) 와 동일하게 **최소 diff** 를 따른다.

## 2. 요청 범위 해석 (스코프)

에이전트는 사용자 문장에서 **스코프** 를 먼저 판별한 뒤, 그 범위 안에서만 수정한다.

### 2.1 좁은 스코프 — 「해당 오브젝트만」

다음과 같이 **특정 대상**만 언급한 경우:

- 「로고 위치 옮겨줘」「검색창 크기 줄여줘」「버튼 오른쪽으로」
- 「이 카드만」「헤더 로고만」「Mova 로고 왼쪽 상단」

**허용**

- 지시된 요소(및 위치 지정에 필요한 **직접 부모 래퍼**, 예: `absolute`용 `div`)만 변경

**금지**

- 같은 헤더 안 **다른** 요소(검색, 네비, 로그인)의 배치·크기 변경
- 모바일/데스크톱 **다른 브레이크포인트** 레이아웃을 임의로 통합·분리
- 다른 페이지(`mova-header` vs `/mova` 랜딩)까지 「일관성」 명목으로 동시 수정 — **각 페이지를 같이 고치라고 한 경우만** 예외

### 2.2 넓은 스코프 — 「영역·전체 구조」

다음처럼 **범위가 넓다**고 명시한 경우:

- 「전체적으로 이쁘게 구조를 바꿔줘」
- 「헤더 레이아웃 다시 잡아줘」「이 페이지 UI 전면 개편」
- 「모바일·데스크톱 헤더 통일해줘」

**허용**

- 명시된 **페이지·영역·컴포넌트 트리** 안에서 레이아웃·구조 재배치
- 그에 따른 반응형·간격·정렬 조정

**여전히 금지 (넓은 스코프라도)**

- 요청과 무관한 **신규 기능** (API, DB, 새 라우트, 새 앱 카드 추가)
- 요청 영역 밖 파일의 임의 리팩터

### 2.3 애매할 때

- 스코프가 불명확하면 **좁은 스코프** 로 가정하고, 넓히기 전에 사용자에게 확인한다.
- 「위치 옮겨줘」만 있고 대상이 없으면 **무엇을** 옮길지 질문한다.

## 3. 판별 예시

| 사용자 요청 | 올바른 범위 | 하지 말 것 |
|-------------|-------------|------------|
| Mova 로고 왼쪽 상단 | 로고(＋필요 시 위치용 래퍼)만 `absolute`/`flex` 조정 | 검색·로그인 행 합치기, `mova-header` 전체 개편 |
| 검색창 작게, 로그인 옆 | 검색 input 너비·높이, 우측 그룹 flex | 로고 위치 변경, 네비 행 삭제 |
| `/apps` Mova 카드 이미지 | `app-museum-card` Mova 분기·이미지 에셋 | 카탈로그 6개 구조 변경, 헤더 수정 |
| 헤더 전체 깔끔하게 재구성 | `mova-header`·랜딩 헤더 등 **명시된** 헤더 파일 | 백엔드·채팅 API 추가 |

## 4. 구현 체크리스트 (PR·diff 전)

1. 변경된 **모든 파일**이 사용자 문장과 직접 연결되는가?
2. 「부수적으로」 옮긴 요소가 있는가? → 있으면 되돌리거나 사용자에게 먼저 확인했는가?
3. 새 파일·새 섹션이 있는가? → 요청에 **추가·생성** 이 있었는가?
4. 스코프가 좁은 요청인데 **2개 이상 페이지**를 수정했는가? → 각각 지시됐는지 재확인
